Обработка ошибок

Следующие условия ДОЛЖНЫ соблюдаться: Ошибки на фронт отдаются в принятом нами формате Все сообщения (поле message) локализуются на сервере при отправке на клиент В документации (apidoc) каждого метода описаны все возможные ошибки для этого метода, включая http status code и код ошибки. Необработанные исключения пишутся в лог с уровнем CRITICAL Следующие условия МОГУТ соблюдаться: Выделен общий класс ошибок и остальные ошибки наследуются от него Ошибки разделяются на сервисные ошибки (Exceptions) и ошибки контроллера (Error). В приложении выбрасываются exceptions, которые потом преобразуются в errors и отправляются на клиент. Следующие условия НЕ ДОЛЖНЫ соблюдаться: HTTP status code присваивается ошибке в сервисе

Всевозможные конвеншны на бэкэнде в node проектах.

Не используем require.main.require Конструкция require.main.require позволяет реквайрить модуль относительно корневого модуля. Обычно корневой модуль это app.js, но при запуске скриптов или миграций корневой модуль может поменяться и сломать рекваеры. Не злоупотребляем index.js Index.js позволяет создавать удобные промежуточные модули и реквайрить непосредственно через них. То есть вместо require('models/user'); писать require('models').user; Это удобно, но в такой схеме в память загружаются все модули описаные в index.js (в данном случае вместо модели пользователя загрузятся все модели). Это может вызвать проблемы во вспомогательных скриптах.

Командный код

Чеклист представляет из себя оглавление из книги чистый код. Книга большая и очень полезная, но ее лучше читать по оглавлению. Имя большинства глав – тезис. Если для вас он очевиден, и не подразумевает разночтений, и вы с ним согласны – можно не чиать. В противном случае читать обязательно. Можно не соглашаться, но делать это надо аргументировано. "А мне не нравится" – не аргумент. Примеры согласия или несогласия с каждым из тезисов надо будет суметь показать в личном или эталонном коде.
  • Именование
  • Имена должны передавать намерения программиста
  • Избегайте дизинформации (контр-пример)
  • Исопльзуйте осмысленные различия
  • Используйте удобопроизносимые имена
  • Выбирайте именная удобные для поиска
  • Избегайте схем кодирования имен
  • Избегайте мысленных преобразований
  • Имена классов и методов
  • Избегайте остроумия
  • Выберите одно слово для каждой концепции
  • Избегайте каламбуров
  • Использутей имена из пространства решения
  • Использутей имена из пространства задачи
  • Добавьте содержательный контекст
  • Не добавляйте избыточного контекста

  • Многие смешивают следующие принципы в одну кучу: 'имена должны передавать намерения программиста', 'правило одной операции', 'один уровень абстракции на функцию', 'избавьтесь от побочных эффектов'. Обратите внимание: это всё разные понятия. Очень важно найти границу между ними.
